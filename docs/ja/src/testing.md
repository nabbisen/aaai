# UI/UX テスト仕様書

本ドキュメントは `aaai-gui` と `aaai` CLI の手動テストケースを定義します。  
リリース前検証を行うテスター向けです。

CLI テストは自動実行できます。

```sh
cargo test -p aaai-cli -- --test-threads=1
```

以下の GUI テストケースは手動で実行してください。

---

## 環境セットアップ

```sh
cargo build --release -p aaai-cli -p aaai-gui

# テスト用ディレクトリを準備
mkdir -p /tmp/aaai-test/{before,after}
echo 'port = 80'   > /tmp/aaai-test/before/config.toml
echo 'port = 8080' > /tmp/aaai-test/after/config.toml
echo 'v1' > /tmp/aaai-test/before/version.txt
echo 'v2' > /tmp/aaai-test/after/version.txt

aaai snap --left /tmp/aaai-test/before --right /tmp/aaai-test/after \
          --out /tmp/aaai-test/audit.yaml
```

## 8. CLI スモークテスト

```sh
# 監査 — Pending のため exit 2
aaai audit --left /tmp/aaai-test/before --right /tmp/aaai-test/after \
           --config /tmp/aaai-test/audit.yaml --no-history

# reason を記入して再実行 — PASSED で exit 0
sed -i 's/reason: .*/reason: "テスト変更"/' /tmp/aaai-test/audit.yaml
aaai audit --left /tmp/aaai-test/before --right /tmp/aaai-test/after \
           --config /tmp/aaai-test/audit.yaml --no-history

# ダッシュボード
aaai dashboard --left /tmp/aaai-test/before --right /tmp/aaai-test/after \
               --config /tmp/aaai-test/audit.yaml

# リント
aaai lint /tmp/aaai-test/audit.yaml
# 期待: exit 0、"No issues found"

# 履歴統計（過去の実行履歴が必要）
aaai history --stats
```

## 10. 視覚検証（RFC 017）

上記の受け入れ基準は「何をテストするか」を定める。RFC 017
（`rfcs/proposed/017-visual-verification-harness.md`）は
「テストしたことをどう記録するか」を定める。

リリース前のテスター作業を終えたら、そのリリースで触れた UI/CLI の各 RFC に
対して **視覚検証カード** を作成し、対応する `rfcs/done/<NNN>-<slug>.md` の
末尾に追記する。コピー用のテンプレートは
`docs/templates/visual-verification-template.md` にある。

カードが未記入の RFC の一覧は、リポジトリのルートで以下を実行すると確認できる:

```sh
scripts/list-unverified-rfcs.sh
```

このスクリプトは CI でも情報目的で毎回実行される。上記の受け入れ基準が満たされる
ことと、視覚検証カードが揃うことは別の作業であるが、リリースタグを切る前に
いずれもゼロになっている必要がある。

## 11. i18n キー監査（RFC 018 §3.4）

上記の視覚検証の **静的な対** にあたるチェック。RFC 016 が辿った
「リテラルキーが GUI に出る」失敗パターンを、GUI を起動する前に検出する。
具体的には `t!()` の全呼び出し箇所のキーを `locales/en.yaml` および
`locales/ja.yaml` の登録キーと突き合わせる。

```sh
scripts/check-i18n-keys.py            # 全件レポート
scripts/check-i18n-keys.py --quiet    # サマリー行のみ
scripts/check-i18n-keys.py --strict   # UNUSED でも失敗扱い
```

MISSING (コードで参照されているがロケールに存在しない) または
DIVERGENT (片方のロケールにのみ存在) が 1 件でもあれば exit 1。
CI では **ブロッキング** ステップとして実行される（視覚検証レポーターは
情報目的だがこちらは違う）。

UNUSED (YAML にあるが呼び出されていない) は通常、機能削除に伴う取り残しで、
自動掃除ではなく機能を削除した RFC のスコープで個別にクリーンアップするのが
望ましい。`--strict` を付けない限り、UNUSED 単独ではビルドを失敗させない。

## 12. ABDD 検証 (手動)

設計書 p.8 の Accessible-by-Default Design チェック観点に対応する手動テスト。
チェックリスト本体は `docs/ja/src/abdd-audit.md` にあり、リリースごとに新しい
シートを埋める。以下のケースはそのシートを埋めるために実機担当者が行う手順
を記述したもの。

| # | 手順 | 期待 |
|---|---|---|
| 12-1 | グレースケール表示で Opening → 監査 → インスペクターを通る | ステータスアイコン `✓ ⚠ ✗ ! —` が相互に判別可能 |
| 12-2 | Opening で Tab を 6 回押す | Before カード → After カード → 任意トグル → 任意フィールド → 監査開始 の順にフォーカス移動 |
| 12-3 | メイン画面でツールバーを Tab で巡回 | 開く → 保存 → 監査実行 → レポート出力、スキップなし |
| 12-4 | Before に存在しないパスを入れて「監査を開始」 | 開始ボタン上にバナーが出て、メッセージ行と次の操作行の 2 行を表示 (silent failure なし) |
| 12-5 | インスペクターで不正な正規表現を入力 | フィールド直下に: エラー本文行 ("Invalid regex") + 次の操作行の 2 行を表示 |
| 12-6 | ツールバー / Start / Approve ボタンにホバー | クリック領域の短辺が 44 px 以上 |
| 12-7 | 承認して保存 vs 破壊的操作 (ルール削除) | 隣接しておらず、視覚的重みも明確に異なる |

実機担当者は観測結果を `docs/ja/src/abdd-audit.md` の該当行に転記し、最終的に
シートに「検証済」のマークを付ける。

### v1.0 範囲外

スクリーンリーダー (NVDA / JAWS / VoiceOver / Orca) の相互運用は v1.0 で
検証しない。iced 0.14 が OS のアクセシビリティ API にウィジェットを公開して
いないため。ABDD シートの §7 で「制限事項」として明記される。iced が
ネイティブの a11y フックを実装した時点で別 RFC として再検討する。
