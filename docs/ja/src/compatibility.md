# 互換性方針 (v1.x)

> **位置づけ.** 本ドキュメントは `aaai` が v1.x 系列で保証する互換性
> 契約を定義する。v0.x 系列には適用されない（v0 では破壊的変更は通常
> の運用だった）。

## ここでの「互換性」の意味

`aaai` は [Semantic Versioning](https://semver.org/) に従う。以下の
ように解釈する:

- **パッチ** (`1.0.X`): バグ修正のみ。正しい入力に対する挙動変化なし、
  スキーマ変化なし。
- **マイナー** (`1.X.0`): 新機能の追加。既存の CLI 呼び出し / 設定
  ファイル / GUI ショートカット / i18n キーは変わらず動作する。新規
  フィールドや新規コマンドが増える可能性はある。
- **メジャー** (`2.0.0`): 互換性保証なし。本ドキュメントに記載された
  項目を含め、リネーム・削除・再構成が許される。

v1.x 範囲内（v1.0.0 から v1.99.x まで）では以下の項目は安定。パッチ
／マイナーリリースで破壊されることはない。

## CLI

### サブコマンド名

v1.x で安定のコマンド名:

| コマンド | 用途 |
|---|---|
| `aaai audit` | 監査の実行 |
| `aaai snap` | 現在の差分から監査定義の雛形を生成 |
| `aaai report` | Markdown / JSON レポートの出力 |
| `aaai check` | 監査定義の検証 |
| `aaai lint` | 監査定義の lint |
| `aaai diff` | ファイル差分のローレベル表示 |
| `aaai merge` | 2 つの監査定義のマージ |
| `aaai init` | 新規監査プロジェクトのスキャフォールド |
| `aaai watch` | ファイル変更に応じた再監査 |
| `aaai history` | 永続化された監査履歴の参照 |
| `aaai config` | ユーザー設定の管理 |
| `aaai dashboard` | プロジェクト 1 画面サマリー |
| `aaai completions` | シェル補完の出力 |
| `aaai export` | 監査結果を移植可能なアーティファクトとしてエクスポート |
| `aaai version` | バージョン表示 |
| `aaai exit-codes` | 標準終了コード一覧の表示 |

これらの削除・リネームは v2.0.0 でのみ可能。

### ロングオプション

`--reason`, `--strategy`, `--from`, `--to`, `--format`, `--out`,
`--quiet`, `--verbose`, `--json` などのロングオプションは安定。
新規追加はマイナー扱い、削除・リネームはメジャー扱い。

### ショートオプション

`-f`, `-t`, `-q`, `-v` などのショートフラグは **ベストエフォート安定**。
サブコマンド間でフラグが衝突した場合、ロング形式を正とし、ショート
形式はマイナーリリースで再割り当てされることがある。
そのような再割り当ては必ず CHANGELOG に記載する。

### 終了コード

v1.x で固定の終了コード:

| コード | 意味 |
|---|---|
| 0 | PASSED — 監査完了、すべて許容内 |
| 1 | FAILED — Failed エントリ 1 件以上 |
| 2 | PENDING — レビューが必要な Pending エントリ 1 件以上 |
| 3 | ERROR — 実行時エラーで完了不能 |
| 4 | CONFIG_ERROR — 定義または CLI 引数が無効 |

いつでも `aaai exit-codes` でこの表を確認できる。

### ヘルプ文言

`--help` の文言はマイナー間で変更される可能性がある。ヘルプ出力を
スクリプトで parse することは推奨しない。各サブコマンドの `--json`
（対応するもの）で構造化出力を取得すること。文言変更は CHANGELOG
に記載される。

## 設定ファイル

### `~/.aaai/profiles.yaml`

`(before, after, definition, ignore)` の命名済み組み合わせを保存する。

- `AuditProfile` への **フィールド追加** はマイナー扱い。新フィールド
  は `#[serde(default)]` を持つので古いファイルも読める。
- **削除・リネーム** はメジャー扱い。
- 新しいマイナーバージョンが書いた未知フィールド付きのファイルは、
  古い v1.x で問題なく読める（未知フィールドは無視され、可能なら
  rewrite 時に保存される）。

### `~/.aaai/prefs.yaml`

ユーザー設定（テーマ、ロケール等）。`profiles.yaml` と同じフィールド
方針。

### `~/.aaai/history.jsonl`

監査実行 1 件につき 1 行の JSON オブジェクト。reader は未知フィー
ルドを許容しなければならない。フィールド追加はマイナー扱い。
1 行 = 1 JSON という形式自体は固定。

### `audit.yaml` (プロジェクト監査定義)

aaai と被監査プロジェクトとの中心的な契約ファイル。`version:`
フィールドで完全にバージョン管理される（現状は `version: 1`）。
`version: 1` 配下でのフィールド追加は、古い reader が無視できる
範囲ならマイナー扱い。互換不能なスキーマ移行は `version:` を
インクリメントし、CHANGELOG に記載する。

`aaai check` がこのファイルの公式バリデータ。

## GUI

### キーボードショートカット

RFC 化されたショートカット、v1.x で安定:

| ショートカット | 動作 |
|---|---|
| Ctrl+S | 監査定義を保存 |
| Ctrl+R | 監査を再実行 |
| Ctrl+Z | 最後の承認を取り消し |
| Ctrl+Shift+Z | 選択中の OK エントリを Pending に戻す |
| Ctrl+Enter | 現在のエントリを承認して保存 |
| Ctrl+E | レポートを出力（保存先ダイアログが開く。拡張子でフォーマットを判別） |
| ↑ / ↓ | ファイルツリーの選択を移動 |
| Tab / Shift+Tab | ペインフォーカスを切り替え |
| Enter | インスペクターの理由フィールドにフォーカス |
| / | 検索バーにフォーカス |
| ? | キーボードショートカット一覧の表示 / 非表示 |
| Escape | 開いているオーバーレイを閉じる、または現在の選択を解除 |

新規ショートカット追加はマイナー扱い。削除・再マッピングはメジャー
扱い。

### ペイン構造

3 ペイン構成のメイン画面（ファイルツリー / 差分ビューア / インスペ
クター）、上部の toolbar、下部のアクションバーは v1.x の契約に
含まれる。ペインの並び替えや削除はメジャー扱い。シート経由で
オプションのペインを追加するのはマイナー扱い。

### i18n キー

`error.*` / `banner.*` / `empty_state.*` / `relative.*` /
`opening.*` / `inspector.*` / `toolbar.*` / `toast.*` などの
名前空間配下のキー識別子は安定。組み込み利用者、新規言語翻訳者は
これらを安定識別子として利用できる。

- 新しいキーの **追加**（新機能や新エラーケース用）はマイナー扱い。
- **リネーム・削除** はメジャー扱い。文言を変更したい場合は、キー
  はそのままで値だけ変える。
- 翻訳済み値の **文言変更** は破壊的ではないが、非自明な変更
  （例: バナーメッセージの書き直し）は CHANGELOG に記載する。

### テーマ

Light / Dark テーマは安定。テーマの描画ディテール（色合いの微調整
など）はマイナー間で変化する可能性がある。

### ドラッグ&ドロップ

Opening 画面は v1.x でフォルダのドロップを受け付ける。フォルダ
以外（ファイル）はインラインエラー表示。将来のマイナーリリースで
カード単位の hit-testing 等が追加される可能性はあるが、それは
追加機能なのでマイナー扱い。

## ライブラリ API (`aaai-core`)

`aaai-core` はコア監査エンジン。
[crates.io](https://crates.io/crates/aaai-core) に公開されており、
API ドキュメントは [docs.rs/aaai-core](https://docs.rs/aaai-core)
にある。

workspace 内では `aaai-cli` と `aaai-gui` が標準的な利用者だが、
外部から監査エンジンを組み込みたい場合は `aaai-core` を直接依存
できる。

public API は v1.x で **ベストエフォート安定** — CLI や設定
ファイルと同じ SemVer 解釈を適用する:

- public 項目（関数、型、`#[serde(default)]` を持つフィールド）
  の **追加** はマイナー扱い。
- public 項目の **リネーム・削除** や関数シグネチャの変更は
  メジャー扱い。

public API に影響しない内部リファクタは自由。v1.x の途中で
public API 変更が避けられない場合（例: soundness バグ修正）、
CLI と同じ opt-in → deprecation → 次メジャーのパイプラインに
従い、CHANGELOG に移行ガイドを記載する。

## 破壊的変更が必要になった場合

セキュリティ問題等で v1.x 中に破壊的変更が必要になった場合の
手順:

1. 新しい挙動はマイナーリリースで **opt-in** として導入する
   （フラグ、新サブコマンド、または新規設定フィールドとして）。
2. 古い挙動は最低 1 マイナーバージョン deprecation 警告を出す。
3. CHANGELOG に移行手順を記載する。
4. 古い挙動の **削除** は次のメジャーリリース (v2.0.0) を待つ。

つまり、v1.x の中では「前向きにマイナーを 1 ステップずつ migrate
する」ことで、ある日突然動かなくなる事態は起きない。

## この契約の対象外

- **パフォーマンス**。パッチ／マイナーリリースで実行時間やメモリ
  特性は変化しうる。ベンチマークは契約対象ではない。
- **派生アーティファクトのファイルシステムレイアウト**。キャッシュ
  ファイル、スクラッチディレクトリ、中間レポート形式の構造は
  マイナー間で変化しうる。
- **エラーメッセージの文言**。構造（message + hint, 終了コード,
  JSON `error.code`）は安定だが、人間可読のテキストはリリース
  ごとに改善される可能性がある。
- **GUI のピクセル単位の見た目**。レイアウト、間隔、色、アニメ
  ーションはマイナー間で進化しうる。契約対象は上記「ペイン構造」
  に列挙された構造要素のみ。
- **スクリーンリーダーとの相互運用**。v1.0 範囲外。ABDD 監査シート
  §7 に制限事項として記録。GUI ツールキット（iced）がアクセシ
  ビリティ機構を公開した時点で別 RFC で扱う。
- **en / ja 以外の言語**。v1.0 は英語と日本語のみ。新規ロケール
  の追加はマイナー扱い。

## v1.0.0 より前のバージョン

v0.x には互換性保証は一切ない。v0 系列は v1.0 への開発過程で
あり、過程で発生した破壊的変更は CHANGELOG に記録されている。
v0.19.0 からアップグレードする場合は v1.0.0 のリリースノートの
移行手順を参照のこと。
